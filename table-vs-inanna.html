When you need to put rows on a screen, you choose where the cleverness lives. These two projects sit at the extremes.

The viewer — lean on the grid

netex-json-browser drops a batteries-included grid straight into the page. SchemaTable.tsx hands a hardcoded COLUMNS array to MUI X DataGrid, and the library does the heavy lifting — virtualization, pagination, column resize, sort headers — for free. The repo writes almost no table machinery; it spends its effort on one table done well: RoleBadge cell renderers, a Descr. info-icon tooltip column, client-side search + role filter (displayRows), and URL state sync (?def=Name) so any definition is a shareable, back-button-aware deep link.

One purpose, one table. The grid is a dependency, not a design problem.

The blueprint — own the abstraction

Inanna refuses to bind itself to any single table. Instead of a grid library it ships a generic, config-driven data view: GenericDataViewPage reads a per-entity *ViewConfig.ts and assembles the UI from plain MUI core parts — DataTableHeader, DataPageContent — wired by hooks like useDataViewTableLogic and useDataViewSearch over a shared Order / sorting type. Declare a config for a new entity and you get a table and a MapLibre map view, no new components. The cost of that generality: the demo domain still leaks into the generic layer (hardcoded 'ParentStopPlace' checks), which issue #5 and OPEN_QUESTIONS.md exist to scrub.

The takeaway

Neither is wrong — they answer different questions. A viewer over a single generated artifact should rent a powerful grid and move on; that is exactly what this repo does, and the payoff is deep-linkable, low-maintenance browsing. A blueprint meant to be forked into many domains must own its table so the abstraction, not the column list, is the reusable unit. Inanna pays in indirection now to make the tenth entity as cheap as the first. Pick the bet that matches your blast radius.
